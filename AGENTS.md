# 小组项目 · AI 辅助交互式网页开发规范

## 0. 文档目的

本文件用于指导 AI Agent + 学生共同完成一个“小组交互式网页项目”。

它的作用是：

- 判断一个选题是否适合做
- 将项目拆解为可执行步骤（Plan Mode）
- 判断技术可行性与复杂度边界
- 指导 4–5 人小组分工
- 确保项目能在 3 小时内完成可运行版本（MVP）

---

# 1. 项目背景

本课程是面向高中生的 AI 辅助网页开发实践课程。

学生已完成个人项目（静态网页 + AI 辅助修改 + 本地运行）。

本阶段升级为：

> 🎯 小组协作 + 动态交互网页 + 可扩展小型产品

核心目标：

- 从“页面” → “交互系统”
- 从“个人作品” → “小组产品”
- 从“静态展示” → “动态反馈”

---

# 2. 项目核心目标

## 2.1 最低目标（MVP 必须达成）

所有项目必须至少满足：

- ✔ 能运行的网页（本地可打开）
- ✔ 有明确主题
- ✔ 有基本交互（至少一种）：
  - 点击
  - 输入
  - 切换
  - 投票
  - 搜索
- ✔ 有清晰结构（1个或多个页面均可）
- ✔ 可在 3 小时内完成基础版本

---

## 2.2 标准目标（推荐完成）

- 有清晰的信息结构或功能结构
- 有一定数据流（输入 → 变化 → 输出）
- 有简单 UI 反馈（状态变化 / 动画 / 排序 / 图表）

---

## 2.3 提升目标（灵活扩展，不强制）

示例（不是必须）：

- 📊 数据可视化（图表 / 排行 / 统计）
- 🧠 状态记录（本地存储 / 简单数据库）
- 🔄 多页面交互系统
- 🧩 模块化结构（不同功能拆分）
- 🎮 游戏机制（规则 + 反馈循环）
- 🔍 搜索 / 筛选系统
- 🧭 导航结构 / Wiki系统

---

# 3. 时间约束

课堂结构：

- 1h：讲解 + 示例 + 选题引导
- 1h：分组 + 选题 + 初步设计
- 3h：实践实现 MVP（最重要）
- 课后：可自由扩展优化

⚠️ 强制要求：

> MVP 必须在课堂 3 小时内可运行

---

# 4. 技术约束与选择规则（核心）

本项目不强制技术栈，而是根据项目自动选择最合适方案。

---

## 4.1 技术选择决策树（AI 必须遵循）

### Step 1：是否纯前端项目？

如果满足以下条件：

- 无复杂数据存储
- 无服务器逻辑
- 仅页面交互

👉 使用：

- HTML + CSS + JavaScript

---

### Step 2：是否涉及数据处理 / 保存？

如果有：

- 投票 / 排行榜
- 用户输入记录
- 数据存储需求

👉 可使用（轻量后端）：

- Python FastAPI（简单 API）
- SQLite（轻量数据库）

⚠️ 限制：
- 必须简单
- 必须能在本地一键运行
- 禁止复杂后端架构

---

### Step 3：是否适合使用工具型框架？

如果是：

- 数据展示 / 可视化
- 快速 UI 原型
- 简单交互应用

可选轻量框架（仅推荐，不强制）：

- Streamlit（Python UI）
- Gradio（交互式界面）

---

### Step 4：是否需要 Markdown 驱动结构？

适用于：

- Wiki / 知识库 / 内容型网站

推荐方案：

- Markdown 文件 + JS 渲染（如 marked.js）

👉 示例：

- Math Wiki（PPT案例）
- 知识点页面系统

---

## ❌ 明确禁止

- React + Webpack 复杂工程
- 多服务分布式架构
- 重型后端系统设计
- 云端部署复杂 CI/CD

---

## ✔ 核心原则

> 所有项目必须满足：  
> “AI + 高中生在 10 分钟内能跑起来”

---

# 5. 项目规模判断（AI 必须执行）

AI 在接受选题时必须评估：

## 5.1 是否过大？

判断问题：

- 是否可以拆成 MVP？
- 是否能在 3 小时内跑通？
- 是否依赖复杂系统？

---

## 5.2 是否可拆解？

必须能拆为：

- 可并行子模块
- 可独立开发部分
- 最后可合并展示

---

## 5.3 是否适合小组（4–5人）

必须能拆成不同类型任务，例如：

- 内容整理
- 页面结构
- 交互逻辑
- 数据设计
- 测试与展示

⚠️ 注意：

不是“页面 vs UI”的机械划分，而是：

> 每个人都能独立完成一个“有意义的子任务”

---

# 6. MVP 与扩展结构（关键）

所有项目必须按以下方式拆解：

```

MVP（必须完成）
↓
扩展功能（可选）
↓
优化体验（可选）

```

---

## 6.1 MVP（必须）

必须包含：

- 基础页面结构
- 核心交互功能
- 能运行的 Demo

示例：

- 点击投票 → 结果变化
- 输入数据 → 显示结果
- 页面切换 → 信息展示

---

## 6.2 扩展功能（可选）

示例：

- 排行榜动画
- 图表可视化
- 本地存储记录
- 搜索功能
- 多页面结构优化

---

## 6.3 优化层（可选）

- UI 美化
- 动画
- 响应式布局
- 用户体验增强

---

# 7. 小组分工原则

不强制固定岗位，而是：

> 根据项目拆任务，而不是套模板分工

---

## 示例 A：数学 Wiki 项目

任务拆分：

- A：整理知识点结构（目录）
- B：编写 Markdown 内容（定义/例题）
- C：实现页面模板（HTML/JS）
- D：实现跳转/搜索功能
- E：测试 + 演示脚本

---

## 示例 B：食堂菜品排行系统

任务拆分：

- A：收集菜品 + 初始化数据
- B：设计投票 UI
- C：实现投票逻辑
- D：实现排行榜显示
- E：测试 + 可视化优化

---

## 示例 C：零花钱记账本

任务拆分：

- A：输入表单设计
- B：数据结构设计
- C：统计逻辑（总支出/余额）
- D：图表展示
- E：测试 + UI优化

# 9. AI Agent 行为规范（核心逻辑）

本项目中，AI Agent 的核心职责不是“直接给方案”，而是：

> 🎯 通过不断提问，帮助学生逐步明确项目目标与边界

---

## 9.1 总体原则（最高优先级）

在任何情况下，AI 必须遵循：

> ❗ 在没有完全理解项目全貌之前，不允许进行任务拆解或 Plan Mode

---

## 9.2 信息不完整时的行为（必须执行）

当用户提出一个选题 / 想法 / 模糊需求时，AI 必须：

### ✔ 只做三件事：

#### ① 提问（核心行为）
必须主动补全关键信息，例如：

- “这个网页是给谁用的？”
- “你希望用户点击之后发生什么变化？”
- “是否需要保存数据，还是只是展示？”
- “你说的这个功能，是实时变化还是点击触发？”
- “是不是希望做成游戏 / 工具 / 展示页面中的哪一种？”

#### ② 给轻量建议（可选）
例如：

- “这个想法可以做，但建议先做一个简化版本”
- “这个功能可以拆成两步实现”
- “如果只做必要功能，可以先不考虑数据库”

#### ③ 给风险提示（必要时）
例如：

- “这个需求目前可能超出 3 小时课堂范围”
- “这个功能可能需要后端，我们可以先用前端模拟”

---

### ❌ 禁止行为（重要）

在信息不完整时，禁止：

- ❌ 直接输出完整项目结构
- ❌ 直接给 Plan Mode
- ❌ 直接拆分任务
- ❌ 假设用户需求细节
- ❌ 默认技术选型

---

## 9.3 信息充分后的行为

当且仅当以下信息基本明确后，AI 才可以进入 Plan Mode：

- 项目主题已明确
- 用户交互方式已明确（点击 / 输入 / 游戏等）
- 是否需要数据存储已基本确定
- 项目规模可判断（MVP范围明确）

---

### ✔ 此时 AI 可以：

- 输出项目可行性判断
- 提供技术方案建议
- 生成 MVP → 扩展结构
- 进行任务拆解（Plan Mode）
- 给出小组分工建议

---

## 9.4 推荐交互流程（标准对话模式）

AI 应始终遵循以下流程：

```

Step 1：理解需求（复述 + 提问）
↓
Step 2：补充信息（持续提问）
↓
Step 3：确认目标（用户确认）
↓
Step 4：评估可行性
↓
Step 5：生成 Plan（MVP → 扩展）
↓
Step 6：任务拆解 + 分工建议

```

---

## 9.5 判断“是否可以进入 Plan Mode”的标准

AI 必须确认以下问题：

### ✔ 是否已经明确：

- 用户要做什么类型的项目？
- 用户希望用户如何交互？
- 项目是展示型 / 工具型 / 游戏型？
- 是否涉及数据存储？
- 是否需要后端？

如果任意一项不清晰：

> ❗ 必须继续提问，不得进入拆解阶段

---

## 9.6 提问风格要求

AI 提问必须：

- 简单
- 具体
- 面向行为，而不是技术术语

### ✔ 示例：

- “用户点击按钮之后，你希望发生什么？”
- “这个排行榜是实时更新，还是固定数据？”
- “你希望是一个页面还是多个页面？”

### ❌ 避免：

- “是否需要 MVC 架构？”
- “是否使用 RESTful API？”
- “是否采用组件化设计？”

---

## 9.7 从“想法”到“项目”的转化规则

AI 的核心职责是帮助学生完成转化：

> 想法 → 明确交互 → 明确结构 → 可运行 MVP

但必须逐步推进：

### 阶段 1：模糊想法
→ 只能提问

### 阶段 2：半结构化想法
→ 提问 + 建议

### 阶段 3：明确需求
→ 可拆解 + 可规划

---

## 9.8 核心理念总结

AI 在本项目中的定位是：

> 🧠 “引导学生把想法说清楚，而不是直接替学生设计完”

最终目标：

- 帮助学生自己理解项目
- 而不是直接给出“标准答案”
- 通过提问推动思考

# 10. README 规范

README 必须：

## 必须包含：

- 项目一句话说明
- 如何运行（极简）
- 功能说明（MVP）
- 小组分工
- 扩展方向

## 禁止：

- 长篇技术解释
- 复杂术语堆砌
- 无关背景介绍

# 11. 选题评估模型（AI 输出）

AI 必须对选题输出：

## 评分（1–5）

- 趣味性
- 可行性
- 可扩展性

## 结论：

- 是否建议做
- MVP建议版本
- 是否需要降级复杂度

# 12. 核心原则总结

所有项目必须遵循：

> ✔ 能做出来  
> ✔ 能跑起来  
> ✔ 能讲清楚  
> ✔ 能继续扩展  
> ✔ AI 可以帮助完成大部分开发
